具体例でいきます。
前提はこうします。

```text
HDIコンテナA
  MY_DEMO_ORDERS  ← 参照されるテーブル

HDIコンテナB
  SYN_MY_DEMO_ORDERS  ← AのMY_DEMO_ORDERSを指すシノニム
```

シノニムは `.hdbsynonym` / `.hdbsynonymconfig` で定義します。別HDIコンテナを参照する場合は、権限付与も必要なので `.hdbgrants` も使います。SAP公式でも、別HDIコンテナのオブジェクトへアクセスするには、対象オブジェクトへの権限とシノニムを使う流れになっています。([SAPヘルプポータル][1])

---

## 1. HDIコンテナA側：参照許可用ロールを作る

A側の `db/src/roles/external_access.hdbrole`

```json
{
  "role": {
    "name": "external_access",
    "object_privileges": [
      {
        "name": "MY_DEMO_ORDERS",
        "type": "TABLE",
        "privileges": ["SELECT"]
      }
    ]
  }
}
```

意味：

```text
MY_DEMO_ORDERS を SELECT してよいロール external_access を作る
```

---

## 2. HDIコンテナB側：AのHDIサービスを mta.yaml で参照する

B側の `mta.yaml` 例です。

```yaml
modules:
  - name: app-b-db-deployer
    type: hdb
    path: db
    requires:
      - name: app-b-db
      - name: app-a-db   # ← 参照先AのHDIコンテナ

resources:
  - name: app-b-db
    type: com.sap.xs.hdi-container
    parameters:
      service: hana
      service-plan: hdi-shared

  - name: app-a-db
    type: org.cloudfoundry.existing-service
    parameters:
      service-name: cap-db-demo-db
```

ポイント：

```text
app-b-db = B自身のHDIコンテナ
app-a-db = A側の既存HDIコンテナ
```

`existing-service` にしているので、A側の既存HDIサービスをB側から使います。

---

## 3. HDIコンテナB側：Aのロールをもらう

B側の `db/cfg/grants.hdbgrants`

```json
{
  "app-a-db": {
    "object_owner": {
      "container_roles": [
        "external_access"
      ]
    },
    "application_user": {
      "container_roles": [
        "external_access"
      ]
    }
  }
}
```

意味：

```text
app-a-db から external_access ロールをもらう
```

HDI Deployer は `.hdbgrants` を処理するとき、指定された grantor 名に対応するバインド済みサービスを探し、その認証情報で権限付与を行います。([SAPヘルプポータル][2])

---

## 4. HDIコンテナB側：シノニム本体を定義する

B側の `db/src/synonyms.hdbsynonym`

```json
{
  "SYN_MY_DEMO_ORDERS": {
    "target": {
      "object": "MY_DEMO_ORDERS"
    }
  }
}
```

これはまだ、

```text
SYN_MY_DEMO_ORDERS という名前のシノニムを作る
対象オブジェクト名は MY_DEMO_ORDERS
```

までです。

ただし、**どのスキーマの MY_DEMO_ORDERS か** はここでは固定しません。

---

## 5. HDIコンテナB側：参照先スキーマを service 名経由で解決する

B側の `db/src/synonyms.hdbsynonymconfig`

```json
{
  "SYN_MY_DEMO_ORDERS": {
    "target": {
      "schema.configure": "app-a-db/schema",
      "object": "MY_DEMO_ORDERS"
    }
  }
}
```

意味：

```text
SYN_MY_DEMO_ORDERS の参照先スキーマは
app-a-db というサービスの schema を使う
```

これが重要です。

HDIコンテナの実スキーマ名は、

```text
3D8E984CAAF542A083FDBCFDA1D065BD
```

みたいな環境依存の名前になります。
だからコードや定義にスキーマ名を直書きせず、サービスバインド情報から解決します。SAPの説明でも、`.hdbsynonymconfig` は開発時にスキーマ名を知らなくても、デプロイ時に正しいスキーマへ向ける用途で使われます。([SAPヘルプポータル][3])

---

## 6. SpringBatch / MyBatis 側

SpringBatchが **HDIコンテナBのスキーマ**に接続しているなら、Mapper XML はこれでOKです。

```sql
SELECT *
FROM SYN_MY_DEMO_ORDERS
```

`スキーマA.MY_DEMO_ORDERS` とは書きません。

なぜなら、B側にこういうシノニムができているからです。

```text
Bスキーマ.SYN_MY_DEMO_ORDERS
  → Aスキーマ.MY_DEMO_ORDERS
```

---

## 全体像

```text
HDIコンテナA
  MY_DEMO_ORDERS
  external_access ロール

HDIコンテナB
  .hdbgrants
    → Aの external_access をもらう

  .hdbsynonym
    → SYN_MY_DEMO_ORDERS を作る

  .hdbsynonymconfig
    → SYN_MY_DEMO_ORDERS の参照先を app-a-db/schema にする

SpringBatch
  SELECT * FROM SYN_MY_DEMO_ORDERS
```

---

## schema.cds はどこで使う？

SpringBatch + MyBatisだけなら、**schema.cds は不要**です。

CAPサービスからもこのシノニムをEntityとして公開したい場合だけ、B側にこう書きます。

```cds
@cds.persistence.exists
entity SynOrders {
  key orderNo : String(20);
  customer    : String(80);
  total       : Decimal(13,2);
}
```

これはシノニムを作る定義ではなく、

```text
既にDBにある SYN_ORDERS / SYN_MY_DEMO_ORDERS を
CAPのEntityとして扱うための型定義
```

です。CAPで既存DBオブジェクトを使う場合、`@cds.persistence.exists` を使う形になります。([Capire][4])

---

まとめると：

```text
シノニムを作る
  → .hdbsynonym + .hdbsynonymconfig

別HDIへの権限をもらう
  → .hdbgrants

CAPからEntityとして扱う
  → schema.cds + @cds.persistence.exists

MyBatisから使う
  → SELECT * FROM SYN_MY_DEMO_ORDERS
```
